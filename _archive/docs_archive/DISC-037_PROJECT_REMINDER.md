## 1. 議題概要

### 背景

DISC-035でPROJECT_PROPOSALSフォルダを新設したが、保存されたプロジェクト案のリマインド方法が未定義だった。Humanから以下の提案があった：

1. 開発完了時にCTAとして未実装プロジェクト一覧を表示
2. 保存内容に壁打ち経緯、期待効果、リスクなどの補足情報を追加

### 検討事項

1. リマインドの5W2Hを明確に定義
2. 表示フォーマットの策定
3. 保存内容の拡充

## 3. 議論結果

### 3.1 リマインド機能の定義

#### 議論の論点

- いつ、どこで、誰が、何を、どうやって表示するか
- 毎回表示すると煩わしくないか
- 効率的な管理方法は何か

#### 各エージェントの主要意見

**Kaizen Susumu（改善業務）**:
- 「開発完了時のリマインドは自然な流れ」
- 「『完了』の定義を明確に（git push完了時）」

**Kotoba Todoku（伝え方）**:
- 「全項目表示すると長くなる、要約版を表示すべき」
- 「詳細は要求時に展開する方式」

**Darui Honne（サボり視点）**:
- 「毎回表示はウザい、『後で』オプション必須」

**Chishiki Takuwaeru（ナレッジ）**:
- 「PROJECT_INDEX.mdで一覧管理すると効率的」
- 「個別ファイルを毎回読み込むのは非効率」

#### CEO決定

```
【決定】リマインド機能の5W2H

| 項目 | 定義 |
|------|------|
| When | 開発タスク完了＆git push/commit完了時 |
| Where | Claude Code会話内 |
| Who | Claude Code（自動表示） |
| What | 未実装プロジェクト一覧（要約） |
| How | 要約表示 → 詳細は要求時展開 |
| How much | 毎回（スキップ可能） |
```

### 3.3 保存内容の拡充

#### 議論の論点

- 5W2Hに加えて何を記録すべきか
- 記録負荷と情報価値のバランス

#### 各エージェントの主要意見

**Chishiki Takuwaeru（ナレッジ）**:
- 「議論の経緯、期待効果、リスクを追加すべき」
- 「ステータス、優先度、日付も必須」

**Kage Tsutsuku（Devil's Advocate）**:
- 「記録負荷が増える、テンプレート文言でカバーすべき」

**Kaizen Susumu（改善業務）**:
- 「議論ログへのリンクで詳細参照、重複記述を避ける」

#### CEO決定

```
【決定】保存フォーマットの拡充

既存の5W2Hに加えて：
- 議論の経緯（DISC-XXXへのリンク + 要約）
- 期待される効果
- 懸念されるリスク＆対策
- ステータス（未実装/実装中/完了）
- 優先度（高/中/低）
- 登録日・最終更新日
- 関連タスク
```

```
- 優先度
- 要約（1行）
- 詳細ファイルへのリンク
```

## 4. 補足情報

### 議論の経緯

DISC-035でPROJECT_PROPOSALSフォルダを新設した際、リマインド方法が未定義だった。Humanから「開発完了時にCTAとして表示」「5W2H + 補足情報で風化防止」という提案があり、8エージェントで詳細を議論した。

### 期待される効果

| 効果 | 説明 |
|------|------|
| プロジェクト提案の風化防止 | 定期的にリマインドされることで忘れない |
| 開発の継続性 | 次に何をすべきか明確になる |
| 情報の充実 | 補足情報により意思決定経緯が残る |
| 効率的な管理 | インデックスファイルで一覧把握 |

### 懸念されるリスク

| リスク | 影響度 | 対策 |
|--------|--------|------|
| 毎回表示が煩わしい | 低 | スキップオプションを用意 |
| 記録負荷の増加 | 中 | テンプレート化、リンク参照 |
| インデックスの更新漏れ | 中 | 記録時に必ずインデックスも更新するルール |
