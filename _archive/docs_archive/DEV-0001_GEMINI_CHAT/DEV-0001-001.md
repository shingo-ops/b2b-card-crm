## 5. 解釈の根拠（思考プロセス）

### 5.1 なぜ暖色系デザインにしたか

**思考プロセス**:
1. 「壁打ち」= カジュアルに考えを話す場
2. 堅い雰囲気は話しにくい
3. 温かみのある色 = 暖色系（オレンジ、ベージュ）
4. 専門家グループ（Katachi Egaku、Kyaku Koe）の意見を採用

### 5.2 なぜ分析ダッシュボードを省略したか

**思考プロセス**:
1. MVP段階では「データ収集」が最優先
2. 分析は後からでも追加可能
3. 最小構成原則（DISC-044）に従う
4. Darui Honneの「週1回の分析は自動化が必須」という意見から、分析機能は別途設計すべきと判断

### 5.3 なぜ危機介入を必須にしたか

**思考プロセス**:
1. 仕様書に「危機介入プロトコル」が明記
2. Yasumu Kokoroの「深刻な悩みには専門家へ」という強い意見
3. 法的リスク（Mamori Houki）の観点からも必須
4. **ここは省略してはいけない**と全員が一致
