## 2. データ管理ルール

### 2.1 原本とバックアップ

| 種別 | 場所 | 用途 |
|------|------|------|
| 原本 | ローカル (`docs/02_development/DEV_RECORDS/`) | 高速読み込み |
| バックアップ | GitHub | データ消失対策 |

### 2.2 同期ルール

1. **記録時**: ローカルに記録 → 即座にGitHubにpush
2. **開発開始時**: `git status`で差分確認 → 差分あれば作業中断・報告
3. **差分発生時**: 記録漏れと判断し、原因調査

### 2.3 差分チェックコマンド

```bash
git fetch origin
git status
# 差分があれば「Changes not staged」または「Untracked files」が表示される
```

## 5. 開発記録フォーマット

各記録には以下を含める：

| # | 項目 | 説明 |
|---|------|------|
| 1 | 基本情報 | ID、プロジェクト名、日付 |
| 2 | 解釈率予想 | 開発完了時の自己評価 |
| 3 | 仕様の意図・KGI | Humanの意図をどう理解したか |
| 4 | 開発ステップ | 具体的な作業内容 |
| 5 | 解釈の根拠 | なぜそう解釈したかの思考プロセス |
| 6 | バグ・手戻り | ジャンル、内容、回数 |
| 7 | 人間の指摘 | 指摘・提案・要求の記録 |
| 8 | 差分分析 | 予想と実際の差分の5W2H |
| 9 | 解釈ズレ原因 | パターン分類 |
| 10 | 成功パターン | うまく解釈できた点 |
| 11 | 所要時間 | 開発にかかった時間 |
| 12 | 改善点 | 次回への申し送り |

## 6. 解釈ズレ原因パターン

| パターン | 説明 | 対策 |
|----------|------|------|
| 思い込み | 「こうだろう」と決めつけた | 仕様書を複数回読む |
| 確認不足 | 仕様書を読み飛ばした | チェックリスト使用 |
| 過剰解釈 | 書いてないことを追加した | 最小構成原則を適用 |
| 過小解釈 | 重要な意図を見落とした | 5W2Hで確認 |
| 技術的制約 | やりたかったができなかった | 事前に技術検証 |
| コミュニケーション不足 | 不明点を確認しなかった | 質問する |
