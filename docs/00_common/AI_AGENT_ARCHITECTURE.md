## 2. 設計思想

### 2.1 コアコンセプト：相手本位

```
従来: 機能本位（全員に同じUI・同じ対応）
     ↓
本設計: 相手本位（個人の人格・性格・習熟度に最適化）
```

### 2.2 根本原則

| 原則 | 定義 | 実現方法 |
|------|------|----------|
| **個人最適化** | 全員違う人間として接する | 診断→教育→実務の全データ活用 |
| **継続成長** | エージェントも成長する | PDCA + 会話ログ蓄積 |
| **多角的視点** | 偏った判断を避ける | 14名のバックエンドチーム |
| **透明性** | 意思決定プロセスを記録 | エージェント間議論ログ |

### 2.3 期待成果

| 指標 | 目標 | 根拠 |
|------|------|------|
| 離脱率 | 50%低下 | 個人最適化による居場所感 |
| 成約率 | 30%向上 | 適切なコーチング |
| 教育期間 | 40%短縮 | 個人に合わせた学習設計 |
| 従業員満足度 | 4.5/5.0以上 | やりがい・承認欲求充足 |

### 5.3 トリガー定義

| トリガー種別 | 条件 | 緊急度 | 議論参加者 |
|-------------|------|--------|-----------|
| **パフォーマンス低下** | KPI 20%以上低下 | 高 | 全員 |
| **連続未ログイン** | 3日以上未ログイン | 高 | 全員 |
| **教育進捗停滞** | 7日以上進捗なし | 中 | 専門家+バランサー |
| **定期レビュー** | 週次/月次 | 低 | 全員 |
| **新規ユーザー開始** | 実務開始時 | 中 | 専門家 |
| **成約達成** | 成約時 | 低 | コーチング+ポジティブ |

## 13. 会議モード仕様

### 13.1 概要

AIエージェント間の議論を可視化・実行するための会議モード。CRMダッシュボード内だけでなく、Claude Codeとの壁打ちでも使用可能。

### 13.2 モード定義

| モード | 出力内容 | 用途 | トークン消費 |
|--------|----------|------|--------------|
| **詳細モード** | 全参加エージェントの発言・議論プロセス | 重要な意思決定、学習目的、透明性が必要な場面 | 多 |
| **簡潔モード** | 決裁者の決定事項のみ | 迅速な意思決定、日常的な確認 | 少 |

### 13.3 モード切り替え方法

| ユーザー指示 | 適用モード |
|-------------|-----------|
| （指定なし） | 詳細モード（デフォルト） |
| 「詳細モードで議論して」 | 詳細モード |
| 「簡潔モードで議論して」 | 簡潔モード |
| 「決定だけ教えて」 | 簡潔モード |
| 「結論だけ」 | 簡潔モード |

### 13.4 詳細モード出力フォーマット

```markdown
## DISC-YYYY-MM-DD-XXX: [議題]

### 議論参加者
- **[エージェント名]**: [参加理由]
- ...

### 1. [論点1]

**[エージェント名]**: 「[発言内容]」

**[エージェント名]**: 「[発言内容]」

### 2. [論点2]
...

### 決定事項

| 項目 | 決定内容 | 根拠 |
|------|----------|------|
| [項目1] | [決定1] | [根拠1] |
| ...

**議論結果**: [最終決定の要約]
```

### 13.5 簡潔モード出力フォーマット

```markdown
## DISC-YYYY-MM-DD-XXX: [議題]

**決裁者**: [CEO/CSO等]
**参加者**: [参加エージェント名リスト]（発言省略）

### 決定事項
1. [決定1]
2. [決定2]

### 根拠（要約）
- [主要な根拠1]
- [主要な根拠2]
```

### 13.6 議論の原則

| 原則 | 説明 |
|------|------|
| **エビデンスベース** | 推測ではなく、データ・研究・理論に基づく発言 |
| **5W2H明確化** | 議題の定義を最初に明確化 |
| **多角的視点** | 専門家・バランサー・ガバナンスの意見を収集 |
| **決裁者決定** | 最終決定は決裁者（通常CEO）が下す |
| **反対意見記録** | 少数意見・懸念も記録に残す |

### 13.7 Claude Codeでの使用方法

このフレームワークはClaude Codeとの壁打ちセッションでも活用可能。

**使用例**:
```
Human: 「新機能Xの実装について議論して」
→ 詳細モードで全エージェント発言
Human: 「簡潔モードで議論して」
→ 決裁者の決定事項のみ出力
```

## 14. 議論モード自動選択仕様

### 14.1 概要

議論開始時に4軸評価により最適なモードを自動選択。手動オーバーライドも可能。

### 14.2 評価軸（4軸）

| 軸 | 記号 | 重み | 説明 | スコア範囲 |
|----|------|------|------|-----------|
| 緊急度 | U | 1.0 | 対応期限の切迫度 | 1-5 |
| 重要度 | I | 1.5 | 組織・顧客への影響度 | 1-5 |
| 複雑性 | C | 1.5 | 判断に必要な専門性の数 | 1-5 |
| 影響範囲 | S | 1.0 | 影響を受けるステークホルダー数 | 1-5 |

### 14.3 各軸のスコア基準

#### 緊急度（Urgency）

| スコア | 定義 | 例 |
|--------|------|-----|
| 1 | 期限なし | 改善検討、将来構想 |
| 2 | 1ヶ月以内 | 次期リリース機能 |
| 3 | 1週間以内 | 今週の対応事項 |
| 4 | 24時間以内 | 翌日必要な判断 |
| 5 | 即時対応 | 障害、クレーム対応 |

#### 重要度（Importance）

| スコア | 定義 | 例 |
|--------|------|-----|
| 1 | 影響軽微 | UIの微調整 |
| 2 | 一部影響 | 特定機能の改善 |
| 3 | 中程度影響 | 主要機能の変更 |
| 4 | 大きな影響 | 顧客体験に直結 |
| 5 | 致命的影響 | 事業継続に関わる |

#### 複雑性（Complexity）

| スコア | 定義 | 例 |
|--------|------|-----|
| 1 | 単純 | 単一観点で判断可能 |
| 2 | やや複雑 | 2-3の観点が必要 |
| 3 | 複雑 | 4-6の観点が必要 |
| 4 | 高度に複雑 | 7-10の観点が必要 |
| 5 | 極めて複雑 | 11以上の観点が必要 |

#### 影響範囲（Scope）

| スコア | 定義 | 例 |
|--------|------|-----|
| 1 | 個人 | 特定エージェントのみ |
| 2 | チーム | 1チーム内 |
| 3 | 複数チーム | 2-3チーム |
| 4 | 組織全体 | 全エージェント |
| 5 | 外部含む | 顧客・Human・外部含む |

### 14.4 スコア計算式

```
議論深度スコア = (U × 1.0) + (I × 1.5) + (C × 1.5) + (S × 1.0)

最小値: 5.0
最大値: 25.0
```

### 14.5 モード閾値

| モード | スコア範囲 | 議論回数 | 参加者 | トークン目安 | 品質目安 |
|--------|-----------|----------|--------|-------------|----------|
| クイック | 5.0-9.9 | 1-2回 | 2-4名 | 500-1,000 | 20-30点 |
| 標準 | 10.0-14.9 | 5回 | 5-8名 | 2,000-3,000 | 60-65点 |
| 詳細 | 15.0-19.9 | 8回 | 9-15名 | 4,000-6,000 | 75-80点 |
| 徹底 | 20.0-25.0 | 10回以上 | 16名以上 | 8,000-12,000 | 90点以上 |

### 14.6 緊急オーバーライド

| 条件 | 動作 |
|------|------|
| 緊急度=5 かつ 詳細/徹底 | 標準モードに強制制限 |
| 緊急度=5 | タイムアウト50%短縮 |

### 14.7 手動オーバーライド

#### 指定方法

| 指定方法 | 例 | 動作 |
|----------|-----|------|
| モード名指定 | 「徹底モードで議論して」 | 指定モードを適用 |
| 回数指定 | 「5回議論して」 | 該当モードを適用 |
| 省略時 | 「議論して」 | 自動計算 |

#### 指定コマンド

```
「クイックモードで議論して」    → クイック強制
「標準モードで議論して」        → 標準強制
「詳細モードで議論して」        → 詳細強制
「徹底モードで議論して」        → 徹底強制
「3回で議論して」              → クイック相当
「8回議論して」                → 詳細相当
「議論して」                   → 自動計算
```

#### オーバーライドルール

| 条件 | 動作 |
|------|------|
| 手動指定 > 自動計算 | そのまま適用（より慎重な判断） |
| 手動指定 < 自動計算 | 警告表示後に適用 |
| 手動指定 + 重要度5 | Human承認は維持 |

### 14.8 優先順位

| 優先度 | 選択方法 | 説明 |
|--------|----------|------|
| 1 | 手動指定 | Humanが明示的に指定 |
| 2 | 緊急オーバーライド | 緊急度5の自動制限 |
| 3 | 自動計算 | 4軸スコアによる判定 |

### 14.9 Human介入ポイント

| 条件 | Human介入 |
|------|-----------|
| 重要度=5 | 必須承認 |
| 影響範囲=5 | 必須承認 |
| モード変更リクエスト | 承認推奨 |
| 議論結果の却下 | 必須承認 |

### 14.10 具体例

#### 例1: UIボタンの色変更
```
U=1, I=1, C=1, S=1
スコア = 1×1.0 + 1×1.5 + 1×1.5 + 1×1.0 = 5.0 → クイック
```

#### 例2: 担当者のモチベーション低下対応
```
U=3, I=4, C=4, S=2
スコア = 3×1.0 + 4×1.5 + 4×1.5 + 2×1.0 = 17.0 → 詳細
```

#### 例3: 新規事業戦略の策定
```
U=2, I=5, C=5, S=5
スコア = 2×1.0 + 5×1.5 + 5×1.5 + 5×1.0 = 22.0 → 徹底
```
